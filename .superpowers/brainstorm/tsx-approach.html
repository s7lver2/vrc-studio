¿Cómo funcionan los archivos .tsx de las tools?

Dos arquitecturas distintas — elige la que encaje con lo que tienes en mente

A

Compilado — TSX como fuente, bundle como output

Cada tool tiene archivos .tsx en su repo (vrcstudio-tools). El build.py los compila con Vite/esbuild a un ui.js bundle. La app sigue cargando ese bundle en iframe, igual que ahora — pero el código fuente es TSX moderno con React, Tailwind, etc.

tool/src/App.tsx → build.py (vite) ui.js → registry
App (Tauri) descarga ui.js → iframe + SDK bridge
Pros
  • Tools completamente aisladas
  • Pueden usar cualquier dep npm
  • Sin riesgo de conflictos con la app
  • Funcionan sin recompilar la app
Contras
  • Build step extra por tool
  • No comparten el design system de la app
  • Tamaño de bundle más grande
  • Sandbox de iframe limita APIs
B

Embebido — TSX cargado en el árbol React de la app

Las tools son archivos .tsx que se registran en la app principal. Se compilan junto con ella o se cargan como módulos dinámicos. Comparten Tailwind, el design system, y el contexto de React de la app. No hay iframe.

tool/src/App.tsx → vite dynamic import App React tree
App monta directamente <ToolComponent /> sin iframe
Pros
  • Acceso a Tailwind, iconos, etc.
  • Look & feel idéntico a la app
  • Sin overhead de iframe/bridge
  • TypeScript end-to-end
Contras
  • Tools deben compilarse con la app
  • Deps deben ser compatibles
  • Sin aislamiento — una tool crashea la app
  • Requiere federated modules o similar
RecomendaciónOpción A — TSX como fuente, bundle compilado como output. Da lo mejor de los dos mundos: escribes con React/TypeScript moderno, pero las tools siguen siendo paquetes descargables independientes. El SDK bridge ya existe y funciona. El cambio es solo en cómo se construyen las tools, no en cómo se cargan. Con Vite + library mode, el bundle incluye React minificado y es completamente portable.
